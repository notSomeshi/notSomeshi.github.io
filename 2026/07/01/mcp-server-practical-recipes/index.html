Model Context Protocol (MCP) 深度指南：连接你的 AI 与现实数据

#Anthropic #MCP #数据连接

引言：打破 AI 的信息孤岛

尽管大语言模型（LLM）已经掌握了人类的大部分编程知识，但在实际的日常开发中，它们仍然像处在一个断网的“信息孤岛”中：

  • AI 不知道你的本地 PostgreSQL 数据库里有哪些表，除非你手动复制 \d 的输出。
  • AI 无法感知你在 Jira 上的 Bug 描述，除非你截图上传。
  • AI 无法直接读取 Figma 的设计图层，只能靠你用文字粗糙地描述。

为了解决这一痛点，Anthropic 于 2025 年底发布了开放协议：**Model Context Protocol (MCP，模型上下文协议)**。它像是一座桥梁，一端连接着 AI 客户端，另一端连接着各类现实世界的数据源。

一、 什么是 MCP？

Model Context Protocol (MCP) 是一个标准化的客户端-服务器协议。

在这个体系中：

  1. MCP Client：通常是 Claude Code、Cursor 或 Roo Code 等 Agent。
  2. MCP Server：轻量级的后台进程，负责暴露特定的 Resources（静态数据）、Prompts（模板）和 Tools（可执行的函数/API）。

通过这一协议，大模型可以“按需”调用服务器暴露出来的工具，例如直接运行 SQL 查询、读取 Jira 票据或检索本地文件。

  • “读取 figma 页面上指定组件的 CSS 属性，并在我们的前端代码中实现它。”

三、 AI 是如何通过 MCP 进行闭环 Bug 修复的？

有了 MCP，AI 修复线上生产 Bug 的流程变得极其丝滑与闭环：

  1. Bug 触发：监控系统发现数据库中有报错，或者用户在 Jira 上提交了 Issue。
  2. AI 介入：Agent 启动，首先通过 jira MCP 服务读取该 Issue 的详细堆栈信息。
  3. 数据检索：Agent 使用 postgres MCP 服务，查询与报错用户相关的库表 schema，甚至安全地执行只读查询获取少量脱敏的报错日志。
  4. 自主定位与修复：结合代码库上下文与数据库实际状态，Agent 定位到是因字段类型溢出导致的报错。
  5. 代码修改与验证：Agent 自动修改代码，运行本地单测，并通过 postgres MCP 运行 migrations 脚本修改测试库结构。
  6. 状态回写：测试通过后，Agent 提交代码，并通过 jira MCP 自动将该 Issue 状态更新为“已修复（Resolved）”，并贴上修改的 Git Diff 摘要。

整个过程中，人类开发者只需作为最终的 Reviewer，极大地提升了生产力的上限。
