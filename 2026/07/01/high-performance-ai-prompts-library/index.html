如何撰写黄金级 AI Prompt：从“做个功能”到“系统级交付”

#Prompt Engineering #提示词工程 #开发效率

引言：别让模糊性毁掉你的 AI 编程效率

在与 AI 协同编程的过程中，许多开发者经常陷入一种困境：

“我让 AI 帮我写一个‘好看的登录页面’，结果它写出来的代码不仅样式奇丑无比，而且连基本的表单校验都没做。AI 根本不好用！”

这种现象被称为 Vibe Coding 的模糊词陷阱。AI Agent 具有强大的执行力，但前提是它必须拥有清晰的边界和输入。如果你给出的指令充满了“好看的”、“鲁棒的”、“简单的”这种主观性极强的词汇，AI 只能基于其概率模型进行猜测，导致最终交付物与你的预期大相径庭。

一、 ICCE 黄金法则

为了从“随便做个功能”升级到“系统级交付”，我们需要在撰写提示词时严格遵循 ICCE 黄金法则。一个优秀的系统级 Prompt 应该包含以下四个核心部分：

  1. **Input (输入)**：明确告诉 AI 要处理的数据、代码段或具体的模块。
  2. **Context (上下文/背景)**：解释这个功能运行在什么环境里，依赖哪些组件，以及业务目的是什么。
  3. **Constraint (约束/限制条件)**：明确列出性能要求、技术栈限制、安全策略以及禁止使用的反模式。
  4. **Examples (示例/Few-Shot)**：给出一个高标准的样本代码或期望的输出格式。

二、 复杂项目的系统级交付链路

当面对大型、跨模块的复杂项目开发时，直接把需求丢给 AI 往往会导致代码库被改得支离破碎。推荐使用以下三阶段交付法：

阶段 1：调研与可行性分析 (Research Session)

在新开的干净 session 中，先让 AI 作为一个资深的系统架构师，对整个项目的可行性进行调研。

  • Prompt 示例：“我现在想在现有系统中加入 WebAuthn 指纹登录功能。请分析我们的前端（Next.js）与后端（Express）架构，列出需要新增的库、潜在的安全风险以及推荐的数据库 Schema 改动，暂时不要写代码。”

阶段 2：输出技术 Spec (Spec Generation)

在调研结果基础上，与 AI 进行多轮讨论，细化所有的技术细节，并让 AI 输出一份详尽的 IMPLEMENTATION_SPEC.md。

  • Spec 内容应包含：文件修改清单、新增依赖、核心逻辑流程图、以及明确的测试用例。

阶段 3：开干净的 Session 执行 (Clean Execution Session)

拿到 Spec 后，在终端中执行 /clear 清除历史上下文，或者新开一个干净的会话。将 IMPLEMENTATION_SPEC.md 作为背景上下文喂给 Agent：

  • Prompt 示例：“请根据 IMPLEMENTATION_SPEC.md 的内容，一步步执行代码修改。每修改完一个模块，请立即运行对应的测试，确保没有破坏现有功能。”

通过这种“调研 -> 写 spec -> 干净执行”的隔离式链路，可以把复杂项目的交付成功率提升数倍，彻底告别上下文污染。
